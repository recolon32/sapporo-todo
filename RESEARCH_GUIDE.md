# 課題調査・サイト構築ガイド

このサイト（札幌市の課題地図）に**課題を追加するときの調査手順**と、
**他都市版を作るときのベース**をまとめたものです。
このファイルごと複製すれば、別の都市でも同じ枠組みで組めます。

---

## 0. サイトの考え方（前提）

- 個別の事件ではなく「放置すると都市の維持が揺らぐ**構造的課題**」を扱う。
- **生成AIによる評価版**。特定の政党・立場ではなく、平等で優先的な判断軸で整理する。
- 内容は**事実 / 解釈 / 提案**に分離する。出典（できれば一次情報）を必ず添える。
- 提案には**トレードオフ・反対論も併記**する。
- 数値・出典は時点で変わる前提（「ゆらぎ」がある）。断定しすぎない。

## 1. データモデル（`data/issues.json` の1課題）

| フィールド | 必須 | 内容 |
|---|---|---|
| `id` | ◯ | 一意のID。`<カテゴリ接頭辞>-<連番2桁>`（例 `snow-01`）。画像ファイル名にも使う |
| `category` | ◯ | `categories.json` の `id` と一致させる |
| `kind` | 任意 | `"plan"` を指定すると「計画・施策」として課題と区別表示（点線枠・バッジ）。通常の課題は省略 |
| `title` | ◯ | 課題の見出し |
| `urgency` | ◯ | 緊急度 1–5（整数） |
| `severity` | ◯ | 深刻度 1–5（整数） |
| `timeframe` | ◯ | `今すぐ` / `5年以内` / `10年以内` / `世代単位` のいずれか |
| `summary` | ◯ | 一行要約（カードに表示） |
| `background` | 任意 | 構造的な背景の深掘り（詳細ページ用） |
| `data` | 任意 | 主要数値の配列 `[{ "label": "", "value": "" }]` |
| `facts` | ◯ | 事実（裏付けのある記述）の配列 |
| `interpretation` | ◯ | 解釈（なぜ構造問題か） |
| `proposals` | ◯ | 提案の配列（トレードオフも書く） |
| `team` | 任意 | 検討体制・メンバー。**実在の体制（出典付き）＋AI想定の追加メンバー**を併記 |
| `aiNote` | 任意 | AI・技術がどれだけ役立つか。**貢献度（小/中/大）＋具体**を書く |
| `deepDive` | 任意 | 「深掘り — 市民の声と答え」。Q&A形式の配列 `[{ "q": "", "a": "", "sources": [{ "title": "", "url": "" }] }]`。→ 詳しくは3.5章 |
| `related` | 任意 | 関連課題の `id` の配列（双方向に張ると図がきれいになる） |
| `plans` | 任意 | 関連する市の計画・制度 `[{ "title": "", "url": "" }]` |
| `sources` | ◯ | 出典 `[{ "title": "", "url": "" }]` |

**インフォグラフィック画像**：`images/<id>.png`（任意）。置けば詳細ページに自動表示、無ければ自動非表示。

## 2. 採点・分類の基準

### 緊急度 urgency（いつ手を打つべきか）
- 5：すでに被害・支障が出ており、今すぐ対応が必要
- 3：数年内に対応しないと選択肢が狭まる
- 1：長期的に備えればよい

### 深刻度 severity（放置した時の打撃の大きさ）
- 5：都市の維持・市民の生命に関わる
- 3：生活・財政に無視できない影響
- 1：影響は限定的

### timeframe（顕在化の時間軸）
`今すぐ` / `5年以内` / `10年以内` / `世代単位`

> 進行中の前向きな取り組み（軽度）は、urgency・severity を低め（例 1〜2）にして整理する。

## 3. 課題1件の調査手順

1. **テーマ設定**：構造的課題か確認（個別事件・短期の政争は除外）。
2. **一次情報を探す**。出典の優先順位：
   1. 市の公式（計画書・統計・公式サイト）
   2. 都道府県・国の資料
   3. 報道（事実確認のため複数ソース）
   4. その他（研究・専門メディア）
3. **事実 / 解釈 / 提案に分離**して書く。数値は「時点」を明記。確認できないものは断定を避ける。
4. **検討体制 `team`**：実在の委員会・所管部局（出典付き）を先に書き、AIが想定する「加えるべきメンバー像」を続ける。
5. **AIノート `aiNote`**：AI・技術の貢献度（小/中/大）と、何にどう効くかを具体的に。効かない場合は正直に「小」と書く。
6. **関連課題 `related`** を結ぶ。
7. **インフォグラフィック**（任意）：Gemini等で生成し `images/<id>.png` で保存。
8. **検証**（→ 5章）→ `updates.json` に1行 → 公開。

### 検索クエリのコツ
- 「<都市名> <テーマ> 課題 計画」「<都市名> <テーマ> 統計 推移」「<都市名> <テーマ> 値上げ／延伸／不足」など、行政の計画名・数値が出る語を足す。
- 公式PDF（`計画`『概要版』など）と報道をセットで当たると、事実と背景が揃う。

## 3.5 「深掘り — 市民の声と答え」（`deepDive`）の作り方

`deepDive` は、その課題に対して**市民が実際に抱きやすい疑問・不満・素朴な問い**を「Q」として立て、
それに**事実と出典に基づいて誠実に答える**「A」を付けるQ&Aセクションです。
すべての課題に付ける必要はなく、**関心が高い・誤解が生じやすい・賛否が割れるテーマ**で特に効果があります。

### ねらい
- 上の「事実／解釈／提案」が"整理された説明"なのに対し、`deepDive` は**生活者の目線の問い**に答える場。
- 「なぜ高いのか」「無駄では？」「結局どうすれば」といった**素朴だが核心的な問い**を逃がさない。
- SNSや報道で見かける指摘・批判を、感情的な対立にせず**事実で受け止める**ための受け皿。

### Q（問い）の見つけ方
1. **SNS・コメント・報道の声を拾う**：X・ニュースのコメント・市民意見などで、その課題に繰り返し出る疑問や不満を集める。
2. **賛否・誤解の分かれ目を探す**：「○○は無駄なのでは」「△△すれば解決では」という単純化された主張は、良いQになる。
3. **当事者の不安を言語化する**：料金値上げ・延伸の遅れなど、生活に直結する不安を一人称の問いにする。
4. 1課題あたり **2〜4問**が目安。多すぎると読まれない。

### A（答え）の書き方
- **事実で答える**。印象論や「べき論」で終わらせず、数値・制度・経緯を根拠にする。
- **問いを否定しない**。「それは誤り」ではなく「こういう事情がある／一部はその通りで、ただし…」と受け止める。
- **トレードオフを正直に**。「下げられない固定費がある」「やれば別の負担が増える」など、簡単でない点も書く。
- **断定しすぎない**。確認できない部分は「要確認」「諸説ある」と明示（サイト全体の"ゆらぎ"方針と同じ）。
- 各Aには可能な限り **`sources`（出典）** を付ける。ここが他のセクションと違い、**Q&A単位で出典を持てる**点に注意。

### 記載フォーマット
```json
"deepDive": [
  {
    "q": "雪が少ない年でも、なぜ予算はあまり下がらないの？",
    "a": "除雪は降ってから動くだけでなく、企業のJVが24時間体制で待機する固定費が大きい。少雪でも待機・人件費・機械維持の費用はかかるため、出動が減っても総額は下がりにくい。",
    "sources": [
      { "title": "札幌市 除雪費の内訳資料", "url": "https://..." }
    ]
  }
]
```

> 実装メモ：`deepDive` があると詳細ページに「深掘り — 市民の声と答え」が自動表示される（Q/Aがカード形式、出典付き）。
> コードを変える必要はなく、`issues.json` に項目を足して課題ページを再生成すれば反映される（4章・6）。

## 4. 課題を追加する実務手順

1. `data/issues.json` に1件追加（テンプレは7章）。`category` は `categories.json` の `id`。
2. `related` は既存の `id` を参照。双方向に張ると関係図が見やすい。
3. **整合チェック**（5章のコマンド）を実行。
4. `data/updates.json` の先頭に更新を1行追加：
   ```json
   { "date": "YYYY-MM-DD", "text": "○○の課題を追加。" }
   ```
5. 画像があれば `images/<id>.png` を置く。
6. **課題ページを再生成**：`python3 tools/build_issue_pages.py`
   - 各課題の静的ページ `i/<id>.html`（クリーンURL＋課題別OGP）を作り直す。
   - トップ・マップは `i/<id>.html` へリンクするので、**追加・削除したら必ず再生成**する（しないとリンク切れ／プレビュー未反映）。
   - 公開先URLを変えた場合は `tools/build_issue_pages.py` の `SITE_BASE` を修正してから再生成。
7. ローカルで表示確認 → `git add -A && git commit -m "課題追加" && git push`。

> **URLとプレビューの仕組み**：詳細ページは `i/<id>.html` という普通のURL（`?` を使わない）で、各ページにその課題のOGP（SNSプレビュー用のタイトル・説明・画像）を静的に埋め込む。クローラはJSを実行しないため、内容は生成時に焼き込んでいる。旧 `issue.html?id=` は `i/<id>.html` へ自動転送。
> **CSS/JS を変えたとき**は、各HTML（と `build_issue_pages.py`）の `?v=YYYYMMDD` を新しい値に上げ、課題ページも再生成する。

## 5. 検証コマンド

ローカルサーバ（`file://` だと `fetch` がCORSで失敗するため必須）：
```bash
cd <このフォルダ>
python3 -m http.server   # → http://localhost:8000/
```

JSONと参照の整合チェック（カテゴリ・関連ID・重複・必須欄）：
```bash
python3 - <<'EOF'
import json
from collections import Counter
cats={c["id"] for c in json.load(open("data/categories.json"))}
issues=json.load(open("data/issues.json"))
ids=[it["id"] for it in issues]; idset=set(ids)
print("issues:",len(issues),"unique:",len(idset))
print("dups:",[k for k,v in Counter(ids).items() if v>1] or "none")
print("bad category:",[it["id"] for it in issues if it["category"] not in cats] or "none")
print("bad related:",[(it["id"],r) for it in issues for r in it.get("related",[]) if r not in idset] or "none")
required=["id","category","title","urgency","severity","timeframe","summary","facts","interpretation","proposals","sources"]
print("missing required:",[(it.get("id"),k) for it in issues for k in required if not it.get(k)] or "none")
print("per category:",dict(Counter(it["category"] for it in issues)))
EOF
```

## 6. 別地域版の作り方（マニュアル）

このリポジトリ一式を複製すれば、別の都市・地域版を同じ仕組みで作れます。
コンテンツ（JSON・文言）を差し替えるだけで、トップ・詳細ページ・課題マップは自動で組み上がります。
下の手順どおり進めれば、コードを書かずに新しい地域版を公開できます。

### 6.1 ファイル構成（現状）
```
index.html      … トップ（カテゴリ別カラーフィルタ＋4カラムカード）
map.html        … 課題マップ（関係ネットワーク図＋時間軸マトリクス）
about.html      … サイト説明・編集方針・更新履歴（最新5件＋全件ページへの導線）
updates.html    … 更新履歴の全件ページ
issue.html      … 旧URL（?id=）から i/<id>.html へ自動転送するだけのページ
i/<id>.html     … 課題ごとの静的詳細ページ（自動生成。直接編集しない）
css/  theme.css … 配色・フォントのトークン（テーマ変更はここ）
      base.css  … レイアウト・コンポーネント
js/   data.js   … 共通（配色パレット・データ読込・補助関数）
      app.js    … トップ（フィルタ・並び替え・カード描画）
      map.js    … 課題マップ（力学配置の関係図＋マトリクス）
      analytics.js … Google アナリティクス（公開元ドメインのみで作動）
      a11y.js   … 文字サイズ調整（あ−／あ／あ＋）
data/ categories.json / issues.json / updates.json
images/ <id>.png（任意のインフォグラフィック） / ogp.jpg（SNS共有用OGP画像）
tools/ build_issue_pages.py … i/<id>.html を生成するスクリプト
```

> ※ 詳細ページ `i/<id>.html` は `tools/build_issue_pages.py` が `issues.json` から生成する。**手で編集せず、必ず再生成する**（手順6.4）。

### 6.2 そのまま使える（変更不要）
- `css/`（`theme.css` の変数を差し替えれば配色テーマだけ変更可）
- `js/`（`data.js` / `app.js` / `map.js` / `a11y.js`）
- ページの仕組み・課題マップ（`categories.json`・`issues.json` から自動生成）

### 6.3 差し替えるもの（チェックリスト）
1. **`data/categories.json`** … その地域の実情に合わせてカテゴリを再編成。`id` は英字短縮（課題IDの接頭辞になる）。
2. **`data/issues.json`** … 3章・3.5章の手順で作り直し。出典はその地域・県・国の公式へ。最初は数件でも公開できる。
3. **`data/updates.json`** … 初版エントリ1行にリセット（例：`{ "date": "YYYY-MM-DD", "text": "初版公開。" }`）。
4. **サイト名・文言** … 各HTML（`index.html` / `map.html` / `about.html` / `updates.html`）の `<title>`・`og:title`・`<h1 class="site-title">`、トップの `.site-desc`、フッター。`tools/build_issue_pages.py` 内の同じ文言（タイトル・フッター）も合わせて変更。
5. **`about.html` の本文** … 目的・「日本全体／その地域の課題と照らして」の説明・クレジットを地域に合わせて書き換え。
6. **クレジット・参考リンク** … フッターと `about.html` の「協力・サイト作成」リンク先（X等のSNS）を自分のものに。
7. **`tools/build_issue_pages.py` の `SITE_BASE`** … 公開先URL（例 `https://<account>.github.io/<repo>`）に変更。OGPの絶対URLに使われる。
8. **`js/analytics.js`** … `GA_ID` を自分の測定IDに、`ALLOW_HOSTS` を自分の公開先ドメインに変更。
   - 元のままでも**許可ホスト以外（フォーク・別ドメイン・localhost）では計測が動かない**ので、元サイトに送信される心配はない。
   - 自分で計測するなら必ず差し替える。計測不要なら `ALLOW_HOSTS = []` にするか、各HTML（と生成スクリプト）の `analytics.js` 読み込みを外す。
9. **`images/`** … 中身を空にして README だけ残す。課題ごとの画像は `images/<id>.png`、トップのOGPは `images/ogp.jpg`（アルファなしJPG・約1400×630推奨。`index.html` の `og:image` で参照）。
10. **`css/theme.css`**（任意） … `--color-accent` などの変数を変えると、サイトとOGP・SNSの世界観をその地域カラーに揃えられる。

### 6.4 ビルド・検証・公開
1. **整合チェック**：5章のコマンドでカテゴリ・関連ID・必須欄を検査。
2. **詳細ページを生成**：`python3 tools/build_issue_pages.py`（`i/<id>.html` を作り直す。課題やCSS/JSを変えたら毎回必須）。
3. **CSS/JS を変えたとき**は、各HTMLと `build_issue_pages.py` の `?v=YYYYMMDD`（`VERSION`）を新しい値に上げてから再生成（キャッシュ対策）。
4. **ローカル確認**：5章のサーバを立てて表示確認（`file://` は `fetch` がCORSで失敗するため不可）。
5. **公開**：`git add -A && git commit && git push` → GitHub Pages 等で配信。

### 6.5 別地域へ展開するときの心構え
- カテゴリは「全国共通の構造課題（人口・財政・インフラ・防災 等）」＋「その地域固有の現れ方」で再編成すると、他地域版と比較しやすい。
- 数値・固有名詞は必ずその地域の一次情報に当て直す。**他地域版の文をそのまま流用しない**（"ゆらぎ"の方針に反する）。
- まず少数の課題で公開し、3.5章のQ&Aや更新履歴を足しながら育てる。

## 8. AIに調査を頼むときのプロンプト雛形

> 「【都市名】の構造的課題のうち〈テーマ〉について、公開情報（市・県・国の公式資料と報道）を調べ、
> 次のJSON1件にまとめてください。事実・解釈・提案を分け、数値には時点を、各記述に出典URLを付け、
> 確認できないものは断定を避けてください。`team` は実在の検討体制（出典付き）＋AI想定メンバー、
> `aiNote` はAIの貢献度（小/中/大）と具体策を書いてください。関心が高い・誤解されやすいテーマなら、
> `deepDive` に市民が抱きやすい問い2〜4件と、事実・出典に基づく答えを付けてください（3.5章）。
> スキーマは RESEARCH_GUIDE.md の7章に従います。」

> ※ `team` / `aiNote` / `deepDive` / `plans` / `data` / `background` は任意。不要なら省略してよい。
